**Decision:** approve-with-conditions

**Reviewer:** system-architect (model: claude-opus-4-8)

## Rationale
The plan is structurally sound and the load-bearing decisions are right. **Q1 (state model):** a single page-level machine — `phase: 'safety'|'questions'|'result'` + `selected: Set<ChipId>` + `questionIndex` + `matches = useMemo(evaluate(selected))` — is the correct shape and is *simpler* than today's flat `HeadacheState` + `chipsFromState()`. The engine already consumes `Set<ChipId>`, so collapsing the typed-field state into the chip set directly removes the entire `chipsFromState` translation layer and its known fan-out traps (`orbital` → 2 chips, `continuous-3mo-plus` → 2 chips). Two hard requirements: (a) the cascade-clear logic must *not* be ported — a persistent live differential with a "See result" escape hatch is incompatible with destructive downstream clearing; revisiting an earlier answer must replace that answer's chips and let `useMemo` re-rank, never wipe later answers; keep a per-question chip map so revisits are surgical. (b) "See result" and "Back" both read the *same* `matches` memo the differential reads — single source, not two computations.

**Q5 (treatment-claim migration) is the one real risk and the reason this is conditions-not-approve.** `HeadacheResultList`/`HeadacheManagement` carry ~20 registered `data-claim` dosing surfaces. The mockup replaces inline dosing with "See also" links, so those claims would render on no page. The claims hook (§13.5) only fails on a tagged claim *missing from the registry* — it does **not** fail on a registered claim that renders nowhere. So retiring `HeadacheManagement` silently orphans ~19 of those claims with a green hook — the false-coverage failure §13.3 warns about. One claim (`clinic-headache-moh-gepant-safe`) is also rendered by `MigrainePathway.tsx:1490`, so it survives; the other ~19 do not. This is a clinical-surface migration that must be resolved *before or with* the rebuild, and it is the clinical-reviewer's call where the dosing goes. **Q6 (phasing):** the proposed P0→P3 order is wrong for an all-on-`main`, no-feature-branch repo because P1/P2 each leave `main` half-migrated. Build the entire v4 as a *sibling* `ClinicHeadachePathwayV4.tsx` wired to nothing, land it across as many commits as needed while the live route still renders the old page, and flip the one import in `App.tsx` in the final commit. Every interim commit ships a coherent page; revert is a one-line flip (§14).

State whether the banded panel (a) extends `MapperPanel`, (b) is a named fork with a written consolidation exit, or (c) supersedes both `MapperPanel` and `HeadacheResultList`. Do not introduce a third parallel differential renderer without retiring one.
- **Q3 — `bandPhenotypes` lives in `src/data/headacheBanding.ts`** (sibling of the engine), consumes `PhenotypeMatch[]`, only buckets (may read `matchStrength`/`criteriaMet`/`criteriaTotal`/`definitionallyExcluded`; must not re-sort/re-rank — the engine owns ranking). Own unit tests pinning the cut points independently of the 84 engine tests.
- **Q2 — drift guard between `headacheQuestions.ts` and the engine.** Each answer maps to `ChipId[]` (many-to-one, mirroring the orbital/continuous fan-out — a single-id config would be a clinical regression). Add a test asserting every mapped chip id resolves via `getChip(id)`.
- **Q4 — extract `HeadacheDotMeter`** (presentation-only, `{ filled, total, band }`, no clinical text). **`PathwayBottomDrawer` retired for this pathway** (mockup has no drawer; result is inline) — confirm no shared code assumes the headache page mounts it. `PathwayHeader` stays.

(2) *Mandatory safety strip* on Frame 3 = non-collapsible, always-rendered, including the empty/no-match case — assert presence regardless of match state. (3) The 6-question linear flow never asks aura/vestibular/TAC-detail/chronic-migraine-detail chip groups, so the differential will rank cluster/hemicrania-continua/SUNCT no higher than "partial." Confirm with clinical-reviewer that banding those permanently "Less likely / set aside" under a 6-question flow is acceptable, or that the sequence conditionally expands (the mockup's "~6" hedge + Skip suggest branching). Decide whether `headacheQuestions.ts` supports conditional "ask only if" gating in P0 or defers it — do not leave it implicit.
