# Architect review — PR #<trial-card-meta-split>

**Decision:** approve-with-conditions
**Reviewed:** plan only
**Reviewer:** system-architect (model: Claude Opus 4.7)
**Date:** 2026-06-03

## Rationale
The problem is real and the target is correct: both `TrialsPage.tsx` (line 15) and `QuestionDetailPage.tsx` (line 23) statically import `TRIAL_DATA`, pulling the 928 KB chunk eagerly to read only `legend` plus ~8 short catalog fields. **Option A (build-time codegen) is the right call**, and it should NOT land as a new `trialCardMeta.ts` file — it should regenerate INTO the existing lightweight `trialListData.ts` layer, which already (a) declares `legend?` on `TrialItem` (lines 24–30), (b) deliberately does not import `trialData.ts` (lines 2–4), and (c) already owns the stub-grade fields the two pages synthesize. A standalone `trialCardMeta.ts` would be a third catalog layer beside `trialListData.ts` and `trialCatalogMeta.ts` — a duplication-rubric concern. Folding the generated legend/year/category back into the catalog modules reverts the Phase 6B split that the `legend no longer in trialListData` comments mark, which is the structurally honest outcome: Phase 6B traded bundle weight for a runtime `.map()` enrichment and a per-page stub-synthesis branch; codegen restores the single lightweight source without re-importing the heavy file. **Option B is rejected** under §4 #1 (clinical safety) and §13.1: a mechanical edit across 108 records inside the 13,975-line clinical-surface `trialData.ts`, plus re-pointing the detail page and `RelatedTrialsSidebar`/`TrialChainTimeline`, is a high-blast-radius confident-but-wrong hazard for zero structural gain over codegen. **Option C (manualChunks / per-property tree-shaking) is refuted**: `legend` etc. are object-literal properties on one giant `const TRIAL_DATA`; Rollup cannot tree-shake individual properties out of a single live binding, and a manualChunks boundary still ships the whole module to any static importer. Codegen is the only approach that severs the import edge.

The decisive hazard with Option A is **drift**: a generated artifact silently going stale against `trialData.ts` would ship wrong legend text — a §13.1 semantic-validity failure, not just a metadata one, because `legend` is authored editorial content ("do not derive at render time", trialData.ts ~line 365). The migration is also low-cost to revert (touches 2 page imports + adds a script + a generated file; no schema rename, no clinical-surface edit), so it clears the §6 migration-exit bar without a feature flag, provided the drift guard exists.

## Required follow-ups
- **Generate into the existing catalog layer, not a new file.** Emit the legend + small fields back onto `TrialItem` in `trialListData.ts` (or a generated `trialListData.legend.ts` it re-exports), so the two pages drop their `TRIAL_DATA` import and read a fully-formed `TrialItem`. Do not introduce a standalone `trialCardMeta.ts` (avoids a third catalog layer).
- **Resolve the stub-trial gap explicitly.** `QuestionDetailPage` synthesizes TrialItems for stub trials (BEST, BASICS, STICH, MISTIE III) that live in `TRIAL_DATA` but NOT in `trialListData`. The codegen must emit catalog rows for these stubs too, or the page will lose them. This is the field-level acceptance check that proves the import can be removed.
- **Minimum drift guard (mandatory, makes Option A safe):** prebuild runs the generator; a pre-commit/CI check regenerates in-memory from `trialData.ts` and fails on any diff against the committed artifact. The generator must be the SOLE writer (header banner: "GENERATED — do not edit"). Without this guard the decision reverts to block.
- **Generator must read `trialData.ts` without executing the app bundle** (parse/import the const, project the whitelisted fields). Pin the exact field whitelist in the script so a new heavy field never leaks into the light chunk.
- **Route the clinical portion to clinical-reviewer.** `legend` is authored clinical-claim content on a `-clinical` task. Clinical-reviewer confirms the generated legend text is byte-identical to source and that no claim-tagged field is dropped or transformed in projection. This review covers structure only.
- **QA: assert the import edge is severed.** Add a build-output check (or bundle-analyzer assertion) that the `/trials` and `/trials/q/*` chunks no longer reference `trialData-*.js`. This is the regression test for the whole PR.

## Blocking issues
None — conditional approval. The single condition that would flip this to block if unmet: the drift guard. A generated clinical-content artifact with no staleness gate is a §13.1 semantic-validity hazard and is not approvable.
