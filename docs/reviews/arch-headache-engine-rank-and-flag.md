# Architect review — headache engine rank-and-flag (drop→demote refactor)

**Decision:** approve-with-conditions
**Reviewed:** plan + touched files (`src/data/clinicHeadacheData.ts`, `src/data/clinicHeadacheData.test.ts`, both consumers `src/pages/ClinicHeadachePathway.tsx` and `src/components/pathways/MapperPanel.tsx`, plus the two clinical audits and the Stage One review)
**Reviewer:** system-architect (model: Claude Opus 4.7)
**Date:** 2026-06-04

## Scope

Pre-implementation structural review of the Class E plan to convert `evaluateHeadachePhenotypes` from "drop near-misses" (the definitional drop-gate at L736-743) to "rank-and-flag," with a SUPPRESS-vs-DEMOTE split on criteria. Bundled clinical-logic corrections coupled to the demote refactor: aura-laterality proxy fix (`auraCharacteristicCount` L391-398), migraine-with-aura `aura-A` ≥2-attacks fix, and a new vestibular-migraine migraine-history suppress gate. This review covers STRUCTURE only; the per-criterion clinical classification is gated separately by the clinical-reviewer pre-execution gate (Class E).

## Rationale

The plan is structurally sound and lands the change in the right place: a single pure module behind a strong import boundary, with the demote behavior expressed as an additive output-shape change plus a re-scoping of one existing gate (L736-743). The core decision — replace the blanket `continue` with a SUPPRESS/DEMOTE split — is correct, and the clinical gate's amendment (chronicity thresholds `cm-A`/`ctth-A` and exclusion criteria `tth-D`/`ctth-D` must SUPPRESS, not demote) is the load-bearing structural constraint, because it means a two-value boolean (`definitional`) can no longer carry the model: the criterion role is now three-way (`scorable` / `demote-gate` / `suppress-gate`), and the current single `definitional?: true` flag conflates the latter two. The encoding choice is the whole ballgame for "does a future criterion silently default to the wrong behavior." Recommend the role enum over the additive `suppress?` boolean, because the boolean creates an illegal-state combination that nothing forbids, whereas an enum makes the classification total and the default checkable. The change does not worsen the existing presentation-policy-in-data-layer smell provided the new flag stays pure data (`definitionallyExcluded: boolean` + `exclusionReason?: string` are fine; a `displayIntent`/band word on the match is not). The collision with the queued Stage One screen build is real but bounded and favorable: `MapperPanel` already declares its own local `MapperMatch` type (`MapperPanel.tsx:25-38`) and does not import `PhenotypeMatch`, so the engine's additive fields cannot break it by construction. The risks that remain are all convertible to checkable gates: the enum migration touching every criterion, the test-contract bifurcation (suppress-stays-absent vs demote-now-present), and the dev-time invariant needing strengthening so no phenotype becomes pure-demote. No structural blockers.

## Per-question findings

**1 — SUPPRESS/DEMOTE encoding: use the role enum, not the boolean.** With two independent booleans (`definitional?`, `suppress?`) the space has four states, only three legal, and a criterion authored with neither flag is indistinguishable from a deliberately-scorable criterion — the unsafe default is silent. With a required enum `role: 'scorable' | 'demote-gate' | 'suppress-gate'`, omission is a type error. The enum also reads at the call site as the actual clinical taxonomy. Rejected third option: a centralized `Set<criterionId>` classification table separate from the criterion objects — more error-prone here, because it splits behavior from definition (the same anti-pattern the Stage One review praised the plan for retiring). Keep the role on the criterion object. `Criterion.definitional` is read in exactly one place in executable engine code (L738) and asserted in one test (L607), so the rename is mechanically contained — but it is referenced in prose across ~12 in-file rationale comments and both audits. Do a literal rename (`definitional?: true` → `role: CriterionRole`) rather than adding `suppress?` alongside, so the old word stops appearing.

**2 — Output-shape vs the 66-test contract: stays defensible, but only if the two expectation classes are physically separated into different describe blocks with explicit comments.** The contract imports only from the engine (test:10-18); the demote change is observable purely through engine output, so nothing leaks. Today the suite encodes the old drop semantics in assertions that will invert: test:53-68 (migraine ABSENT with only photophobia — mig-D), test:113-130 and test:132-146 (episodic TTH ABSENT on nausea / photo+phono — but these are `tth-D`, a SUPPRESS criterion, so they must STAY absent), test:648-655 (`fail mig-B → absent` — flips to present-as-probable), test:657-664 (`fail mig-D → absent` — flips), test:700-707 (`fail tth-B → absent` — flips), test:726-733 (`fail cluster-B → absent` — flips), test:833-841 (empty-integration). The danger: a reviewer sees "tests changed" and waves it through without noticing some absences are now correct (suppress) and some are now bugs (demote). Structure as two top-level describe blocks — `SUPPRESS gates: failure drops the phenotype` (holds `tth-D`, `ctth-D`, `cm-A`, `ctth-A`, `vm-A`, indomethacin gate, continuous-contradiction — stay green unchanged) and `DEMOTE gates: failure keeps the phenotype as a flagged near-miss` (holds `mig-B`, `mig-D`, `tth-B`, `ctth-B`, `cluster-B`, `ph-B`, `sunct-B` — each asserts `matchStrength` now `'probable'`/`'partial'`, `definitionallyExcluded === false`, failed criterion in `missingCriteria`). The per-criterion block at test:646-804 is the template; bisect it along the new axis rather than the old `definitional` axis.

**3 — Bundling the three aura/VM corrections: keep aura-laterality bundled (correctly coupled); `aura-A` ≥2-attacks and the VM migraine-history gate are separable but cheap — bundle all three, do not split.** Demoting `migraine-with-aura` will, for the first time, surface `aura-C` in a `missingCriteria` list with its description text (L439), and that description currently tells a clinician "unilateral" satisfies an aura characteristic while the evaluator wires it to `loc-unilateral` (pain laterality, L398). Ship the demote without the laterality fix and you generate a prompt that instructs the clinician to tick the wrong chip — the refactor manufactures the bug surface. The `aura-A` 1-attack fix and the new VM gate touch the same two phenotype objects the refactor already rewrites, so splitting means a second clinical gate over the same lines for marginal isolation benefit. The one thing NOT to treat casually: the chip-vocabulary additions themselves (new `aura-symptom-unilateral` chip, any `attacks-ge-2` chip, the `migraine-history-established` chip) — each new `ChipId` is a feature-vocabulary change the Stage One question-config drift-guard and the data-integrity test (test:300-311) must both cover. Make new chips explicit line items with their own integrity assertions.

**4 — Boundary integrity: the refactor leaves the `displaySection`/`PROBABLE_SECTION_FOR` presentation-policy-in-data smell roughly where it is — acceptable for this PR — but the new flag must NOT carry display intent.** `displaySection` (L780-782) already picks the §X.5 label string for the `probable` strength. The refactor exercises that policy more often but does not deepen the coupling, because `definitionallyExcluded`/`exclusionReason` can and should be pure facts. Hold that line: `exclusionReason` names the failed criterion/gate in neutral terms (e.g. `"chronicity threshold not met (cm-A)"`), never a UI band word or a display sentence. Right division for the screen build: engine emits `matchStrength` + `definitionallyExcluded` + `exclusionReason` + met/missing as data; the page (or `headacheResultCopy.ts` per Stage One follow-up 3) owns every clinician-facing word. Optional opportunistic improvement (do NOT gate this PR on it): stop computing the §X.5 string in the engine and emit a `probableSectionRef` token the copy layer resolves.

**5 — Collision with Stage One: invalidates none of the seven Stage One conditions; two need a one-line restatement.** Conditions 1, 2, 3, 6, 7 are orthogonal to engine output shape — untouched. Condition 5 (`useMemo(() => evaluateHeadachePhenotypes(selected), [selected])`) holds verbatim; the memo's return type gains fields. Condition 4 (defer band words; Stage One renders neutral placeholders) is the one to restate: Stage One must now pass through `definitionallyExcluded` and decide whether suppressed-with-evidence phenotypes render at all — recommend Stage One filters them (`matches.filter(m => !m.definitionallyExcluded)`) and defers grayed-tray treatment to Stage Two, keeping Stage One Class D. Favorable finding: `MapperPanel` carries its own `MapperMatch` (MapperPanel.tsx:25-38, field `section` not `displaySection`, `matchStrength` excludes `'none'`), so the engine's additive fields cannot break the existing primitive. Restatement worth making explicit: Stage One's "engine and its 66-test contract are untouched" framing is now false for this PR — annotate that the engine landed a governed Class E change ahead of it; the contract is re-baselined, not bypassed.

**6 — Dev-time invariant: preserved as written, but MUST be strengthened to "≥1 SUPPRESS-class gate per phenotype," because the current test would pass a pure-demote phenotype that can never be hidden.** Today's invariant (test:605-610, "every phenotype has at least one `definitional` criterion") becomes ambiguous when `definitional` is replaced by a role enum: a mechanical rewrite to "≥1 criterion with role !== 'scorable'" would pass a phenotype whose only gate is a `demote-gate` — exactly the pathological case (nothing can ever suppress it; it surfaces as a near-miss on a single contributing chip with no way to fall out, reintroducing the `vm-A`-class false-positive the definitional mechanism was built 2026-05-27 to kill). Strengthen to: every phenotype has ≥1 criterion with `role === 'suppress-gate'` OR a `hiddenUntilTrial` gate (HC/PH gate via `hiddenUntilTrial`, so the invariant must accept either). Spot-check against the plan: **migraine-without-aura keeps no suppress criterion under the plan (both `mig-B`/`mig-D` demote) — a real gap the strengthened invariant catches.** §1.1 migraine has `mig-A` (scorable), `mig-B` (demote), `mig-C` (scorable), `mig-D` (demote) — zero suppress gates, no `hiddenUntilTrial`. The clinical-reviewer must resolve whether §1.1 needs a suppress gate (it has no chronicity/substrate criterion to serve as one) or whether the `hasPositiveEvidence` floor (L727-729) is deemed sufficient suppression for feature-only phenotypes.

## Required follow-ups

1. **Encode the split as a `role: 'scorable' | 'demote-gate' | 'suppress-gate'` enum replacing `Criterion.definitional`, not as an added `suppress?` boolean.** Make the field required so omission is a type error. *Checkable: `definitional` appears zero times in `clinicHeadacheData.ts` after the change; `role` is non-optional on `Criterion`; tsc fails if any criterion omits it.*
2. **Keep the new `PhenotypeMatch` fields pure data.** `definitionallyExcluded: boolean` + `exclusionReason?: string`; `exclusionReason` names the failed gate in neutral terms and contains no UI band word or display sentence. *Checkable: grep the new fields' assigned values for band words ("Leading"/"Possible"/"Less likely") and result prose — none.*
3. **Bifurcate the test contract along the new axis.** Two top-level describe blocks: SUPPRESS gates (failure → phenotype absent; `tth-D`, `ctth-D`, `cm-A`, `ctth-A`, `vm-A`, indomethacin gate, continuous-contradiction — assertions stay green) and DEMOTE gates (failure → present as `probable`/`partial`, failed criterion in `missingCriteria`, `definitionallyExcluded === false`; `mig-B`, `mig-D`, `tth-B`, `ctth-B`, `cluster-B`, `ph-B`, `sunct-B`). Each inverted assertion carries a comment naming why it flipped. *Checkable: the demote-criteria tests at test:53-68/648-664/700-707/726-733 now assert presence; the `tth-D` tests at test:113-146 still assert absence; both blocks pass.*
4. **Strengthen the dev-time invariant to require a suppression path per phenotype.** Replace test:605-610 with: every phenotype has ≥1 criterion with `role === 'suppress-gate'` OR a `hiddenUntilTrial` gate. *Checkable: the test fails for any phenotype that is pure-scorable+demote with no `hiddenUntilTrial` — which under the current plan flags `migraine-without-aura` (see follow-up 5).*
5. **Resolve the §1.1-migraine no-suppress-gate gap surfaced by follow-up 4 with the clinical-reviewer before execution.** Decide whether `hasPositiveEvidence` (L727-729) is the accepted suppression floor for feature-only phenotypes (and if so, encode that decision so the strengthened invariant accepts it explicitly), or whether §1.1 needs a designated suppress gate. *Checkable: a written clinical-reviewer line item ruling on the floor; the invariant test encodes whichever answer is chosen and passes.*
6. **Cover every new `ChipId` in the data-integrity test and the Stage One question-config drift guard.** The aura-symptom-laterality chip, any `attacks-ge-2`/split-`attacks-lt-5` chip, and the `migraine-history-established` chip are vocabulary changes, not incidental edits. *Checkable: test:300-311 (every `contributingChip` resolves to a real chip) passes with the new chips; each new chip is referenced by at least one criterion.*
7. **Decide and document whether suppressed-with-positive-evidence phenotypes are emitted at all.** The plan says suppress gates "still drop," but `definitionallyExcluded` only has meaning if such phenotypes appear in output. Pick per-gate: (a) suppressed phenotypes are `continue`d and never appear (then `definitionallyExcluded` is only ever `false` on emitted matches — reconsider whether the field is needed), or (b) suppressed-with-evidence phenotypes ARE emitted carrying `definitionallyExcluded: true` so the page can render a grayed "considered and excluded — [reason]" tray. The clinical gate's "silent deletion is the more dangerous failure mode" reasoning argues for (b) for suppress gates that fail on positive contradicting evidence (`tth-D`/`ctth-D`: patient has nausea) but (a) for substrate-absence gates (`vm-A`: no vertigo — nothing to show). Resolve per-gate, not globally. *Checkable: each `continue` site has a comment stating emit-or-drop and why; tests assert the chosen behavior for at least `tth-D` (contradicting evidence) and `vm-A` (absent substrate).*
8. **Route the clinical correctness of the SUPPRESS/DEMOTE assignment and the §1.1-floor decision to the clinical-reviewer pre-execution gate (Class E requirement).** This review covers only the structure carrying the split; the per-criterion classification, the new VM migraine-history gate's clinical necessity, and the follow-up-5 floor ruling are clinical calls the §17.2 gate owns. *Checkable: `docs/reviews/clinical-headache-engine-rank-and-flag.md` exists with decision `approve`/`approve-with-conditions` before execution begins, per §6 Class E and §20.*

## Blocking issues

None. The plan is structurally approvable. The eight follow-ups convert the three real risks — (i) an under-specified two-value encoding silently mis-defaulting a future criterion, (ii) a test contract whose inverted assertions blur suppress-correct absences with demote-bug absences, and (iii) a pure-demote phenotype that can never be hidden — into checkable gates before execution. The §1.1-migraine no-suppress-gate gap (follow-up 5) needs a clinical decision rather than a mechanical fix, but it is surfaced, not latent, once the invariant is strengthened, so it does not block structural approval.
