# Architect review — headache result accordions (Stage One, result-presentation increment)

**Decision:** approve-with-conditions
**Reviewed:** plan only
**Reviewer:** system-architect (model: claude-opus-4-8)
**Date:** 2026-06-04

## Rationale
The plan is structurally sound and correctly scoped: it collapses three stacked result blocks (headline card 713–777, differential ribbon 784–840, multi-diagnosis banner 847–886) into one ranked native `<details>/<summary>` accordion list, in a new presentational component `HeadacheResultList.tsx` that takes `{ matches: PhenotypeMatch[] }` and touches nothing clinical. The boundaries hold — the engine, `chipsFromState`, state shape, questionnaire, the ~13 `data-claim` treatment cards (888–1037), footer, and drawer are all left alone, and the red-flag short-circuit (the `!redFlagActive` guard plus the drawer's non-collapsible "Secondary workup" tier) is untouched, which preserves the SNNOOP10 backstop. On **Question 1 — the key call — build the headache-specific `HeadacheResultList`; do not adopt `MapperPanel`.** Reusing `MapperPanel` looks like the disciplined "use the existing primitive" move but is a net regression here: its local `MapperMatch.metCriteria` is `{id,label}[]` and would drop the `contributingChipLabels` audit trail that the page renders today (1166–1171) and that a clinical-reviewer condition mandated (selection→criterion mapping, page 383); its `missingCriteria` requires a `description` field that `PhenotypeMatch` does not carry (adapter must blank it); and its hardcoded `StrengthChip` wording ("Consistent with Probable / Partial match") has **no chronic-migraine-probable exception**, whereas this page carries a load-bearing rule that chronic-migraine probable must NOT say "Probable" because ICHD-3 §1.5 does not cover §1.3 patients (page 363–369, 792–796, 865, a prior clinical-reviewer condition). Adopting `MapperPanel` therefore forces either a lossy adapter (clinical regression) or an edit to a shared primitive that pulls every consumer and the clinical reviewer deeper into a copy delta — exactly what this increment is trying to avoid. The cost of choosing build-specific is honest duplication: two near-identical accordion renderers (`MapperPanel` + `HeadacheResultList`) and a `CriteriaList` lifted/duplicated out of the page. That duplication is acceptable for this increment **only if it is named with a consolidation exit** rather than left silent (rubric items 1 and 3), which is the single condition below. On the other rubric items: boundary integrity passes (Q2 — `src/components/pathways/headache/` is the right home; it is the folder the held Stage One artifact already designates for headache-specific pathway components, condition 3 there); state locality passes (the component is a pure prop sink, no state lifted); dependency weight passes (no new package — note `MapperPanel` uses `lucide` icons while the current page uses inline `✓`/`○` glyphs, so build-specific also avoids importing an icon dep the result region does not currently use); migration exit passes (two files touched, well under the five-file threshold, trivially revertible by restoring 668–886, no schema/shared-abstraction change, no rollback flag needed). Q4: leaving the inline treatment cards in place and deferring the treatment link-out is structurally acceptable, not half-migrated drift — those cards are the registered `data-claim` surfaces, are a separately-owned clinical increment, and the result/treatment seam is clean (the accordion list ends, the per-phenotype cards begin). Q5: the only things at risk of being lost in the 3-blocks→1-list collapse are (a) the per-match `role="progressbar"` a11y semantics and (b) the verbatim multi-diagnosis / "ICHD-3 allows multiple codes / treat each separately" guidance — both must be carried into the new component, not dropped, which I make explicit conditions below.

## Required follow-ups
- **Name the duplication and its exit (rubric 1 + 3 — required).** `HeadacheResultList` is intentionally a parallel renderer to `MapperPanel`, and `CriteriaList` is intentionally lifted/duplicated from the page. Add a top-of-file comment in `HeadacheResultList.tsx` stating: (i) why `MapperPanel` was not reused (PhenotypeMatch carries `contributingChipLabels` + the chronic-migraine-probable section exception that `MapperMatch` cannot represent); and (ii) the consolidation trigger — when `MapperMatch` is widened to carry `contributingChipLabels` and a section-exception hook, or when a second pathway needs this exact renderer, the two converge. This converts silent divergence into a tracked, intentional fork. Optionally log a one-line `TASKS.md` PARKING LOT / Future-Refactors entry referencing the same.
- **Resolve `CriteriaList` ownership (rubric 3 — required, pick one and state it).** The treatment cards (892, 919, 946, …) still call `CriteriaList` and so does the new component. Either (a) export the existing `CriteriaList` from a shared module both the page and `HeadacheResultList` import (preferred — single source), or (b) keep a private copy inside `HeadacheResultList` and document that the page's `CriteriaList` remains the canonical one. Do **not** silently define a second `CriteriaList` with identical logic and leave both un-cross-referenced — that is the duplication this rubric exists to catch.
- **Carry the progressbar a11y semantics into the summary row (rubric 2 / Q5 — required).** The current headline and ribbon expose `role="progressbar"` with `aria-valuenow/min/max` and a per-match `aria-label` (757–769, 815–827). The compact accordion must not silently drop this — preserve a `role="progressbar"` (or an explicit equivalent) for the criteria-met proportion in each summary row, or state in the plan that the percentage is intentionally demoted and why. Keep the section's `role="status" aria-live="polite" aria-atomic` wrapper intact as the plan says.
- **Preserve the multi-diagnosis guidance verbatim (rubric 2 / Q5 — required).** The "ICHD-3 General Principles allow more than one primary headache code… treat each phenotype separately" caption (836–838) and the additional-consistent-phenotypes enumeration (847–886) are existing reviewed clinical strings. They must be relocated into the new component verbatim (as the plan states) — confirm none is paraphrased or dropped when the standalone banner becomes a list caption. Any wording change here flips the increment to D-clinical.
- **Preserve the chronic-migraine-probable section/label exception (rubric 2 — required).** The new summary-row tag logic must reproduce the `isChronicMigraineProbable` branch (792–796) so chronic-migraine probable renders "Partial"/`displaySection` and never "Probable §1.5". This is a prior clinical-reviewer condition, not a style choice.
- **Restyle, do not re-author, the empty-match fallback (rubric 2).** Folding 668–693 into the component's empty state is fine; the neutral "no clean ICHD-3 match" wording (intentionally not directing to secondary workup) must be relocated verbatim. Adding a `data-claim` here is out of scope and should not be introduced.
- **Routing to clinical-reviewer — conditional, not required.** If every relocated string is byte-identical (verifiable by grep/diff of the moved literals against 668–886), this stays Class D and needs no clinical gate. If any string is reworded — including band-word relabels, which the plan explicitly defers — it becomes D-clinical / Class E and must route to `clinical-reviewer` before execution. State which case applies in the PR.

## Blocking issues
None. Two files touched, boundaries intact, safety path preserved, revert trivial. The conditions convert the one real structural risk — silent duplication of an accordion renderer and of `CriteriaList` — into a named, exit-tracked fork, and pin the four reviewed clinical strings (progressbar semantics, multi-diagnosis caption, chronic-migraine exception, neutral empty-state) so the "verbatim relocation only" claim is checkable rather than honor-based.
