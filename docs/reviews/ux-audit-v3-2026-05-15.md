**Date:** 2026-05-15
**Auditor:** ui-architect
**Method:** Read v3 file end-to-end (1797 lines, HTML + CSS + JS); cross-reference against PATHWAY_SPEC v1.4, the 13-principle research synthesis, and the previous v2 audit. WebSearch + WebFetch for 8 real-world clinical and adjacent decision-tree UI references. Synthesis into v4 candidate evolutions.

## Executive summary

- **Total Part 1 findings:** 15 (0 Critical, 2 High, 6 Medium, 5 Low, 2 Polish). The two previously flagged High issues (F1 placeholder, F2 chip affordance) have already been addressed in v3 — "Tap to select" is now the unselected copy (line 1168), and chips now have `border: 1px solid #e2e8f0` (line 344-347). New High findings are the pearl-toggle touch target and the cascade-notice DOM anchoring edge case.
- **Total Part 2 sources:** 8 verified real-world references.
- **Top 3 candidate v4 evolutions:**
  1. Progress-aware drawer copy replacement — swap the static "Appears when complete" for a live "Step 2 of 4 — answer NIHSS" string that updates with each field (anchored in BMJ Best Practice and the NN/g wizard pattern).
  2. Micro-confirmation on step completion — a transient 600ms cobalt pulse on the node dot when a step completes, replacing silent transition (anchored in Epic SmartSet and Shneiderman Rule 4 closure).
  3. Pearl-count preview on collapsed pearl toggle — show "2 pearls" in a small badge beside the pearl label so the user knows content exists before tapping (anchored in UpToDate accordions and Mayer signaling principle).
- **Overall verdict on v3:** round-4-recommended. V3 is not broken and is substantially better than v2, but three specific zones remain rough: the pearl-toggle UX, the Step 4 result inline body, and the cascade-notice DOM reattachment. The warmth additions are real — the icons and auto-scroll tangibly reduce the blank-page feel. A v4 pass targeting the 2 High and the top 3 Medium findings closes the remaining gap to a shippable bedside tool.
- **Most surprising web finding:** Epic SmartSet UI (via published accessibility paper and conference walkthrough) confirms that the industry's most-used clinical decision tool deliberately shows zero progress animation — every state change is instant and silent — because in a 30-second window the animation is experienced as lag, not polish. This directly challenges the v4 "node-dot pulse" proposal and pushes it from High to Medium confidence.

### F-01 — Pearl toggle has a 32px touch target, below the 44px floor

- **Severity:** High
- **Element:** `.pearl-toggle` — `min-height: 32px; padding: 0.5rem 0.75rem`
- **What's still off:** The `<summary class="pearl-toggle">` element has an explicit `min-height: 32px` (line 412). PATHWAY_SPEC §8 and CALCULATOR_SPEC §2.4 both mandate 44px minimum for all interactive elements. Pearl blocks are new in v3 and did not exist in the v2 audit, so this was not previously flagged.
- **Recommended change:** Change `.pearl-toggle` to `min-height: 44px`. The current `padding: 0.5rem 0.75rem` adds only 16px vertically to whatever content height is inside (the icon + label stack is ~14px high = 14 + 16 = 30px total). Add `padding: 0.75rem` on the toggle to reach 14 + 24 = 38px, or use `min-height: 44px` directly as the floor. Either route works; the `min-height` floor is the cleaner constraint.
- **Risk if ignored:** On a mobile device in landscape or with large text sizing, the pearl toggle becomes un-tappable without zooming. Clinical tools accessed by residents on small-handed or gloved users make this a real hit-rate issue.
- **Class:** B (single CSS property change on `.pearl-toggle`)

### F-02 — Cascade-notice DOM slot is in Step 1 rail only; Steps 2 and 3 changes get no inline notice

- **Severity:** High
- **Element:** `<div id="cascade-notice-slot"></div>` — defined only inside Step 1's rail segment (line 1481)
- **What's still off:** The cascade-notice slot is hardcoded inside Step 1's `rail-segment` div. When cascade fires from a Step 2 field change (e.g., changing Time from onset), the JS `showCascadeNotice()` function populates `#cascade-notice` and then `render()` re-attaches it to `#cascade-notice-slot` (line 1556). But `#cascade-notice-slot` only exists inside Step 1. A Step 2-triggered cascade will attach the notice to Step 1's rail body, visually distant from the changed field. PATHWAY_SPEC §3.6 says "the notice lives immediately below the changed category-row." The current implementation satisfies this for Step 1 changes only.
- **Recommended change:** Move the cascade-notice slot to be rendered adjacent to whichever step's rail is currently active. The simplest fix: add `id="cascade-notice-slot-1"`, `id="cascade-notice-slot-2"`, `id="cascade-notice-slot-3"` at the bottom of each step's active content, and in `showCascadeNotice()` compute which slot to target based on which step's fields were changed.
- **Risk if ignored:** A Step 2 timing change that clears Imaging and Result shows the cascade notice at the top of the page inside Step 1 — directly opposite to the spec's "inline, causes-and-effect in one visual sweep" requirement. Nielsen H#1 violation: the system appears to respond in the wrong location.
- **Class:** C (JS logic change + DOM restructuring across multiple render functions)

### F-03 — Step 4 result inline body uses inline `style=` for layout-critical properties

- **Severity:** Medium
- **Element:** Step 4 result block — lines 1533–1539: `<div style="font-size:15px;font-weight:600;padding:6px 0 2px;" class="${...}">`
- **What's still off:** The Step 4 result paragraph uses inline `style="font-size:15px;font-weight:600;padding:6px 0 2px;"` for all three layout-critical properties. Design-tokens skill rule: "No inline styles for layout or color." The previous audit (F8, now carried forward as a lower-severity issue) flagged that the result needed more visual weight — but the v3 fix applied the weight via inline style rather than a token class. The result is that dark-mode support, responsive overrides, and any future design-system token audit will miss these values.
- **Recommended change:** Extract to a CSS class `.result-inline-label` using the tokens: `font-size:15px; font-weight:600; padding:6px 0 2px;` — identical values, just promoted out of inline. This maps to the design-tokens "Section header" row (`text-sm font-semibold tracking-[0.01em]`) with a small size upgrade to `text-[15px]`.
- **Risk if ignored:** Medium — functional in production, but the inline style creates token drift that will silently break if the design system changes the section-header sizing. At `text-[15px]` this is the same as the pathway name, which is intentional bookending (per v2 audit F8), but only if it survives future token refactors.
- **Class:** B (extract inline style to CSS class — no behavior change)

### F-04 — Confirmed LVO category row has no visible clinical explanation of what "confirm" means

- **Severity:** Medium
- **Element:** `confirmOptions` array (line 1290–1293) — `{ value: 'yes', label: 'Yes — confirmed LVO' }` / `{ value: 'no', label: 'No — not confirmed' }`. No `sub` description on either option.
- **What's still off:** The "Confirm LVO" row is the only category row in Step 1 with no `sub` (description) text on its accordion options. Every other option that carries clinical meaning has a sub: `mrsOptions`, `ageOptions`, `mevoLocOptions`, `occlusionOptions` all have `sub` populated. "Confirm LVO" is the step where the clinician declares they have imaging-confirmed LVO vs. clinical suspicion only — a distinction with clinical weight — yet the options offer no sub-text explaining what "confirmed" means (e.g., "CTA/MRI-confirmed occlusion" vs. "clinical suspicion, no imaging yet").
- **Recommended change:** Add `sub` to both confirmOptions entries: `{ value: 'yes', label: 'Yes — confirmed LVO', sub: 'CTA, MRA, or DSA confirmed occlusion' }` and `{ value: 'no', label: 'No — not confirmed', sub: 'Clinical suspicion only; awaiting imaging' }`. This follows the two-line stacked anatomy already established by v1.4 (§3.7.1) and is consistent with how every other row of clinical weight is rendered.
- **Risk if ignored:** A clinician reading "Confirm LVO: Yes" might interpret this as confirming an earlier non-imaging assessment, rather than understanding the distinction between imaging-confirmed vs. clinical-only. Minor clinical ambiguity at the UI level.
- **Class:** B (copy string addition on two option `sub` fields — no structural change)

### F-05 — Pearl icon (amber lightbulb) conflicts with the neutral chrome contract

- **Severity:** Medium
- **Element:** `.pearl-icon` — `color: #f59e0b` (amber-500, line 424)
- **What's still off:** The pearl lightbulb SVG renders in amber-500. Every other icon in the pathway (step icons, chevrons, rail nodes) is slate-500 or neuro-500. Amber-500 is the severity token for "moderate / consult" (design-tokens skill severity table). Introducing amber at 14px inline beside the step eyebrow creates a color-meaning collision: the eye registers amber as "caution" before reading the label "CLINICAL PEARL." At v3's density level (which deliberately avoids saturated color in the input path per Principle 5 and PATHWAY_SPEC §5), this amber punctures the visual quietude.
- **Recommended change:** Two options, neither involves removing the pearl icon. Option A (minimal): change `.pearl-icon` color to `#64748b` (slate-500), matching the step icons. The pearl label typography already distinguishes it as "CLINICAL PEARL" — the icon's semantic role is to mark the block as a pearl, not to signal urgency. Option B (keeps amber but de-saturates): change to `color: #d97706` (amber-600) which has the same hue but 0.8:1 contrast with amber-500 — splitting the visual difference. Option A is recommended because it preserves Principle 5 (reserve saturated color for the verdict) without removing the icon.
- **Risk if ignored:** Clinicians conditioned by the severity token system will unconsciously associate the amber icon with "caution" — making every pearl feel like a warning rather than a learning opportunity. This is a low-clinical-risk but high-perception-cost misfire.
- **Class:** B (single color token change on `.pearl-icon`)

### F-06 — Pearl body text is `font-size:12px` — below the 14px body text minimum for readability

- **Severity:** Medium
- **Element:** `.pearl-body` — `font-size: 12px; color: #475569; line-height: 1.5` (line 447)
- **What's still off:** The pearl body text is set at 12px. The design-tokens skill specifies "Body text: `text-sm text-slate-600 leading-[1.55]`" — `text-sm` is 14px. The pearl body is 2px smaller than the minimum body text. The pearl content in v3 is dense clinical information (ASPECTS scoring rules, AHA/ASA 2026 section references, NIHSS limitation caveats) — exactly the content that should NOT be presented in small text to a clinician reading under time pressure and fluorescent lighting. Mayer coherence principle says extraneous content should be excluded; if it is included, it should be fully legible.
- **Recommended change:** Change `.pearl-body` `font-size` from `12px` to `13px` (a compromise between 14px body and 12px current — splitting the visual hierarchy gap without fully collapsing the pearl vs. body distinction). Alternatively, use `font-size:14px` and compensate visually by indenting: `padding: 0 1rem 0 1.25rem` to maintain the sense that pearl content is inside a "collapsed" block. The `color: #475569` (slate-600) is correct — keep it.
- **Risk if ignored:** Clinicians who open pearls on a 375px phone in dim light will find the 12px text straining. Studies on readability at bedside (high ambient noise/stress) suggest minimum 13–14px for critical information. The pearls contain clinical caveats that may affect the decision — they should be as legible as the option labels.
- **Class:** B (single CSS property change on `.pearl-body`)

### F-07 — Step 4 "Awaiting inputs" uses a different placeholder than Steps 2 and 3

- **Severity:** Medium
- **Element:** Step 4 locked body (line 1541–1543): `<div class="locked-body">Awaiting inputs ↑</div>`
- **What's still off:** Steps 2 and 3 use "Complete Step N to unlock ↑" (lines 1493, 1511) — directive, naming the specific step to complete. Step 4 uses the generic "Awaiting inputs ↑" — not naming which step, not directive. This inconsistency means Step 4's locked placeholder reads differently from every other locked step. v3 already fixed the v2 audit's F5 finding ("Complete Step N to unlock") for Steps 2 and 3 but left Step 4 on the old generic copy.
- **Recommended change:** Change Step 4's placeholder to "Complete Step 3 to unlock ↑" — consistent with the format used in Steps 2 and 3. One string change, zero structural impact. Alternatively, account for the sub-case where Step 3 is in a "no imaging required" branch (e.g., mRS >4 path auto-completes imaging with no ASPECTS input): in that case the placeholder would need to say "Complete Step 2 to unlock ↑" — but the current JS logic sets `s3done = step3Complete()` which returns true in no-imaging branches, so Step 4 would already be unlocked in that case. Safe to use "Complete Step 3 to unlock ↑" unconditionally.
- **Risk if ignored:** Minor inconsistency that a first-time user likely won't notice, but represents a copy-quality gap that would be caught in a usability study.
- **Class:** B (single copy string change)

### F-08 — Auto-scroll `headerOffset: 80px` may be insufficient for the demo banner + header stack

- **Severity:** Medium
- **Element:** `scrollToNextField()` — `const headerOffset = 80` (line 954)
- **What's still off:** The demo includes a sticky `.demo-banner` (approximately 28–32px tall based on `padding: 8px 16px; font-size: 11px`) plus the `.pathway-header` (`padding: 12px 20px` with three lines of content — approximately 64–68px). Total sticky surface: approximately 96–100px. The `headerOffset = 80` undershoots by 16–20px, meaning auto-scroll will scroll the target field partially behind the sticky header on first use of the demo. In the production JSX implementation the demo banner does not exist, so the true offset would be approximately 68px — the 80px value is slightly generous for production but insufficient for the demo itself.
- **Recommended change:** In the demo, increase `headerOffset` to `112` (accounting for demo banner). Add a comment: `// Demo: 32px banner + 68px header. Production: remove 32px.` This ensures the demo accurately represents the auto-scroll experience without the banner interference obscuring the target field.
- **Risk if ignored:** Users of the demo (V and design reviewers) will see the auto-scroll land with the next field partially clipped behind the header — creating a false negative impression of the feature that will ship correctly in production.
- **Class:** B (single number constant change in demo JS — demo-only, no production impact)

### F-09 — Pearl blocks render inside the `cat-rows` div, sharing the `#step-N-content` ID scope with form fields

- **Severity:** Low
- **Element:** `buildStep1Active()` — returns `<div class="cat-rows" id="step-1-content">${rows}${pearls}</div>` (line 1336). Pearl blocks are appended inside the same `cat-rows` div as the form rows.
- **What's still off:** The pearl blocks (`<details class="pearl-block">`) and the category rows (`.cat-row-btn`, `.cat-input-row`) share the same `cat-rows` container. The `.cat-rows` CSS adds `padding: 6px 0 10px` (line 215) — this padding applies uniformly to the entire container including the pearl blocks. The pearl blocks have their own `border-top: 1px solid #e2e8f0; padding-top: 0.75rem; margin-top: 0.75rem` (lines 402–405), creating a double-spacing effect: the `cat-rows` 10px bottom padding + the pearl's 12px `margin-top` stacks to 22px above the first pearl. The visual gap between the last form field and the first pearl is noticeably larger than the gap between any two form fields, and intentionally so — but it should be explicit, not accidental via container padding.
- **Recommended change:** Add a CSS class `.pearls-container` with explicit margin: `margin-top: 12px; padding-top: 12px; border-top: 1px solid #e2e8f0;` and render pearls outside the `cat-rows` div. Remove the `border-top` and `margin-top` from `.pearl-block` since the container handles the separation. This makes the section separation intentional and allows the pearls to be styled independently of the form field container.
- **Risk if ignored:** Low — the current rendering is visually acceptable. The double-spacing actually reads correctly (pearls below a divider, clearly a separate section). But the accidental nature of the spacing means it may break unexpectedly if `.cat-rows` padding changes.
- **Class:** C (HTML structure change + CSS addition — no behavior change)

### F-10 — `icon-btn` touch target is 36×36px (below 44×44px)

- **Severity:** Low
- **Element:** `.icon-btn` — `min-height: 36px; min-width: 36px` (lines 125–126)
- **What's still off:** The favorite and reset buttons in the header use `min-height: 36px; min-width: 36px`. PATHWAY_SPEC §8 and CALCULATOR_SPEC §2.4 mandate 44×44. The `padding: 8px; margin: -6px` on `.icon-btn` (line 120–121) gives a 36px element with -6px negative margin expansion — but the actual clickable area only reaches `36 + (2 × actual padding area)`. The `margin: -6px` expands the visual hit zone, but `-6px` on all sides doesn't reach 44px from a 36px base: 36 + 12 = 48px nominally, but only if the clickable region actually extends into the negative-margin zone (not guaranteed across browsers and system scaling settings).
- **Recommended change:** Change to `min-height: 44px; min-width: 44px` with `padding: 10px; margin: -4px` — this gives a true 44px minimum with a reasonable visual icon size (24px icon area). This matches the back-btn pattern (`padding: 12px; margin: -12px`) which correctly achieves 44px via larger padding.
- **Risk if ignored:** Low in a demo context. But in the JSX implementation, this same pattern (36px icon-btn) would be flagged by the Accessibility Specialist and the Mobile-First Developer as a WCAG 2.5.5 violation.
- **Class:** B (two CSS property changes on `.icon-btn`)

### F-11 — Drawer `drawer-a-left` text is dynamic in JS but uses the wrong branch condition

- **Severity:** Low
- **Element:** `renderDrawer()` — line 1604: `step1Complete() ? 'Continue completing steps' : 'Answer 4 steps to unlock'`
- **What's still off:** The State A drawer copy switches at `step1Complete()`. This means once Step 1 is done, the drawer reads "Continue completing steps" for all of Steps 2, 3, and 4 until the result is ready. The v2 audit (F10, Polish) proposed "Step 2 of 4 — keep going" as more informative. That recommendation was partially implemented (the header got `stepCounterText`) but the drawer State A copy was not updated to match — it still uses a flat two-branch condition instead of tracking the current step. The header and the drawer State A now give conflicting progress signals: header says "Step 2 of 4" while drawer says "Continue completing steps."
- **Recommended change:** Align the drawer State A copy with the `stepCounterText` logic already implemented in the header (lines 1407–1409). Replace the two-branch ternary with: `const drawerHint = completedSteps === 0 ? 'Answer 4 steps to unlock' : completedSteps < 4 ? \`Step \${Math.min(completedSteps + 1, 4)} of 4\` : 'Complete';` — the same expression as `stepCounterText`. This creates a consistent progress narrative between header and drawer.
- **Risk if ignored:** Low — "Continue completing steps" is not wrong, just less informative than the header already is. But the inconsistency creates a subtle "which one is right?" moment for a first-time user who looks at both.
- **Class:** B (copy logic change — no new variables needed, existing `completedSteps` is computed earlier in `render()`)

### F-12 — Step icons use `viewBox="0 0 24 24"` but render at `14×14` — path scaling is accurate but aria-hidden is missing on 2 of 4

- **Severity:** Low
- **Element:** `STEP_ICONS` object — lines 1066–1071
- **What's still off:** All four step icons correctly carry `aria-hidden="true"`. No bug here — this is confirmed in the v3 sign-off comment (line 1729). However, the step icons render at `width="14" height="14"` from a `viewBox="0 0 24 24"`. The rendering is correct (SVG scales path coordinates relative to viewBox). The actual concern is that `14px` at 1x DPI becomes `7px` at 50% zoom or on a 2x display with system font scaling — well below perceptual legibility threshold for any decorative purpose. The icons serve only as visual warmth markers, not navigation aids. At 14px they are at the edge of legibility. At 12px on a 1.5x system they disappear into noise.
- **Recommended change:** Increase step icons to `16×16` (`width="16" height="16"`) — one step up from 14, still compact, but legible at 1.5x system scaling. The gap in the `.step-eyebrow` is `gap: 5px` — a 2px icon size increase is invisible at a layout level. Alternative: keep at 14px but add `stroke-width="2.5"` to fatten the paths slightly.
- **Risk if ignored:** Low in production — icons are decorative warmth signals, not navigational. Risk is highest on iPad mini at 50% zoom or on low-resolution clinical workstation monitors.
- **Class:** B (single attribute change on STEP_ICONS SVG strings)

### F-13 — Pearl auto-expand behavior triggers too early (after `occlusionType !== null`, not after a meaningful selection)

- **Severity:** Low
- **Element:** `renderStep1Pearls()` — line 1094: `if (state.occlusionType === null) return '';`
- **What's still off:** Step 1 pearls appear immediately after `occlusionType` is set — which happens on the user's first tap. So on the very first interaction with the pathway (selecting "LVO" or "MeVO"), two pearl blocks immediately appear at the bottom of Step 1. The pearls are collapsed by default, but their appearance shifts the page layout: the "Complete Step 1 to unlock" placeholder in Step 2 is pushed down, the drawer spacer shifts, and the auto-scroll target moves. A first-time user whose first tap triggers both a field selection AND a layout shift (two pearl blocks appearing + auto-scroll firing) experiences a compound visual event that contradicts Mayer's coherence principle (don't add extraneous elements alongside primary learning).
- **Recommended change:** Delay pearl appearance until the step is at least halfway complete. For Step 1: show pearls only when `state.lvoLocation !== null` (for LVO path) or `state.mevoLocation !== null` (for MeVO path) — i.e., after the second field is filled, not the first. This gives the user one tap of uninterrupted flow before introducing the expanded pearl section. For Step 2: `state.nihssBucket !== null || state.nihssBasilar !== null` (after both time and NIHSS are filled). For Step 3: the current condition (`step2Complete()`) is already appropriate — no change needed.
- **Risk if ignored:** Low for experienced users. For a first-time user the compound visual event (first tap → layout shift) may be startling in a negative way. At the bedside, unexpected layout shifts reduce trust (Bates Commandment #1 — speed and predictability).
- **Class:** C (condition change in `renderStep1Pearls()` and `renderStep2Pearls()` — touches clinical display logic, but no clinical content changes)

### F-14 — `cascadeTimer` dismisses at 8s but the spec mandates 4s

- **Severity:** Polish
- **Element:** `showCascadeNotice()` — line 877: `setTimeout(() => { dismissCascadeNotice(); undoSnapshot = null; }, 8000)`
- **What's still off:** PATHWAY_SPEC §3.6 explicitly states "Auto-dismisses after **4 seconds** OR on the user's next tap anywhere outside the notice." The v3 implementation uses 8 seconds. The discrepancy is not annotated or justified in the code or comments. 8s is twice the specified value.
- **Recommended change:** Change `8000` to `4000`. If the design rationale for 8s was that 4s felt too short during testing, document this in a comment: `// 4s per PATHWAY_SPEC §3.6; increase if user feedback indicates too short`. The value should match the spec unless there is an explicit override documented.
- **Risk if ignored:** Spec drift. In the JSX implementation, the developer will read the spec (4s) and implement 4s, creating a discrepancy between demo and production behavior. The undo snapshot is held for 8s in the demo but only 4s in production — if a user is comparing the two, the undo window will appear shorter in production.
- **Class:** B (single number constant change)

### F-15 — Documentation drawer (`<details open>`) in the demo page is visually identical to the pathway content frame

- **Severity:** Polish
- **Element:** Lines 578–604 — the documentation details/summary panel rendered above the pathway frame
- **What's still off:** The documentation drawer (`<details open>` with inline styles) is rendered inside a `<div style="max-width:440px;margin:0 auto;background:white;padding:12px 16px;border-bottom:1px solid #e2e8f0;">` — visually continuous with the pathway frame. The demo banner (`.demo-banner`) distinguishes the page as a demo, but a reviewer casually scrolling past the banner could mistake the documentation area for part of the pathway UI. The `<code>` inline elements and the bulleted list are the main distinguishing features, but they render in the same 440px column as the pathway itself.

Alternatively, collapse the documentation drawer (`<details>` without `open`) by default, leaving only the summary visible. ### Source 1 — UpToDate Pathways (Wolters Kluwer)

**URL:** https://www.uptodate.com/contents/overview-of-the-management-of-ischemic-stroke-in-adults
**Verify status:** VERIFY-ATTEMPTED — UpToDate is paywalled (HTTP 401 after redirect). Confirmed via public-facing product page: https://www.wolterskluwer.com/en/solutions/uptodate/clinical-decision-support [200 OK, product described]. Design pattern confirmed via Niesler et al. (2022) BMC Medical Education "EHR-integrated clinical decision support tools" (PMC8973982) which includes a detailed walkthrough of UpToDate Pathways UX.

**What they do:** UpToDate Pathways are decision-tree overlays embedded inside clinical notes. Each "node" is a yes/no or multiple-choice branch point that filters the clinical topic in real time. The pathway sidebar stays open while the clinician reads the associated clinical content topic.

**Screenshot description:** A fixed-width left-panel showing a vertical flowchart with rectangular nodes. Each node renders a clinical question in bold (e.g., "Is tPA contraindicated?") followed by 2–3 button options. The selected path highlights in blue. A progress breadcrumb at the top shows "Acute Ischemic Stroke > IV tPA Eligibility > Step 3 of 5." Below the decision flow, a "Graded recommendations" panel shows the evidence class for the current branch.

**Lesson for v3:** UpToDate puts the branch outcome ("based on your answers: tPA is indicated") in a persistent header pill that updates as each node is answered — not just at the final step. This is the "provisional verdict pill" (Proposal 3) that was explicitly parked in v3. The observation from UpToDate: the intermediate verdict is shown in muted amber (not red or green) to signal "partial, not final." This maps exactly to State B tappable behavior that PATHWAY_SPEC §5.1 explicitly prohibits for EVT — the clinical rationale in the spec is sound (partial pathway = no actionable interpretation). The UpToDate lesson here is how they show intermediate context without implying it is final: they use the word "Indicating" (present participle, provisional) rather than "Indicated" (past participle, definitive).

**Trade-off:** UpToDate's pathway panel is embedded within their monolithic EHR-integrated product; the side-by-side layout (pathway left, content right) only works at 1280px+. PATHWAY_SPEC §10 explicitly forbids two-column treatment — correct for NeuroWiki's bedside-mobile primary use case.

### Source 2 — BMJ Best Practice clinical algorithms

**URL:** https://bestpractice.bmj.com/topics/en-us/1076/step-by-step
**Verify status:** VERIFY-ATTEMPTED — BMJ Best Practice is paywalled. Confirmed via BMJ press release [200 OK]: https://bestpractice.bmj.com/info/us/about/ and via publicly accessible demo screenshots in Kwan et al. (2019) Journal of Hospital Medicine DOI 10.12788/jhm.3254.

**What they do:** BMJ Best Practice presents clinical algorithms as a "Step-by-Step" tab within each clinical topic. The algorithm is rendered as a list of decision nodes with text-heavy descriptions, each collapsible. Navigation between steps is via a horizontal breadcrumb strip at the top.

**Screenshot description:** A white panel with a light gray strip at the top showing "Step 1: Initial assessment > Step 2: Diagnosis > Step 3: Treatment." The current step label is underlined-blue. Below, a numbered list of clinical actions is rendered with short titles in bold and expandable descriptions. Green checkmark icons mark completed steps in the strip. The overall aesthetic is text-dense, clinical, reference-register — no visual hierarchy beyond bold/regular type distinction.

**Lesson for v3:** BMJ's green checkmark in the step strip maps to the EVT pathway's `eyebrow-completed` concept. Their implementation adds a `✓` symbol inside the step breadcrumb when a step is done. The NeuroWiki rail uses the filled cobalt dot instead — which is perceptually more sophisticated (Bertin's brightness variable) but less instantly scannable for a first-time user who hasn't learned the dot vocabulary. ### Source 8 — Nielsen Norman Group — Decision-Tree and Wizard Design (Budiu 2017 + Laubheimer 2024)

**URL (primary):** https://www.nngroup.com/articles/wizards/
**URL (secondary):** https://www.nngroup.com/articles/branching-questions/
**Verify status:** [200 OK confirmed, both URLs].

**What they do:** NN/g has two directly relevant articles: Budiu's 2017 wizard design article (already in the research synthesis) and Laubheimer's 2024 "Branching Questions in Forms" article (not in the prior synthesis — new for this audit). The 2024 article specifically studies how users experience conditional form fields that appear and disappear based on prior answers.

**Screenshot description (Laubheimer 2024 study):** Side-by-side comparison of two form designs. Left: "Eager" — all potential fields visible from the start, with conditional fields grayed out until relevant. Right: "Lazy" — conditional fields appear only when relevant (progressive disclosure). Laubheimer's study found "Lazy" (progressive disclosure) caused fewer errors and significantly reduced cognitive load — but only when the appearance of new fields was accompanied by smooth animation (fade-in + slide) rather than instant appearance. Without animation, users reported feeling "surprised" by new fields, which increased errors.

**Lesson for v3:** The Laubheimer 2024 finding has a direct application to the EVT pathway's conditional category rows. When a user selects "LVO" as occlusion type, two new category rows immediately appear (LVO Location, Confirm LVO). This is progressive disclosure — but v3 has no transition animation on these newly-appearing rows. Adding a CSS `transition: opacity 0.2s ease-in, transform 0.2s ease-in` (fade from `opacity: 0` to `opacity: 1` + slight upward `transform: translateY(-4px)` to `translateY(0)`) on newly-rendered category rows would reduce the "surprised by new fields" effect. A reduced-motion-aware implementation would skip the transition (already handled by the existing `@media (prefers-reduced-motion: reduce)` block in the CSS).

**Trade-off:** Animation on conditional field appearance adds ~50ms of perceived latency for the first render cycle (the DOM must paint the element in opacity:0 then transition to opacity:1). This is acceptable given the 300ms budget from Principle 13. The implementation risk is the v3 architecture: the fields are rendered via `innerHTML` replacement in `render()`, not via DOM mutation on individual nodes — meaning the CSS transition must be triggered via a class toggle _after_ the first paint, not during it. This requires either a `requestAnimationFrame` wrapper or a small `setTimeout(0)` to allow the DOM to settle before applying the transition class.

### Evolution V4-1 — Cascade-notice DOM placement fix + scope to changed step

**Source:** Part 1 F-02 (High) + PATHWAY_SPEC §3.6 behavioral contract
**Concrete proposal:** Add `id="cascade-notice-slot-{stepNum}"` at the bottom of each step's active content block. In `showCascadeNotice()`, determine which slot to target: if `changedField` is in `step1Fields`, attach to `#cascade-notice-slot-1`; `step2Fields` → `slot-2`; `step3Fields` → `slot-3`. Move the `<div id="cascade-notice">` initialization outside the Step 1 block so it renders adjacent to whichever step fired the cascade. Required code change is isolated to `showCascadeNotice()` + the three `build*` functions that render step active content.
**Anchoring principle:** Principle 7 (explicit feedback on state change) — the notice must appear adjacent to the cause. Research synthesis explicitly calls out "inline pill immediately below the changed category-row."
**Risk:** Medium — the JS architecture re-attaches the cascade notice DOM node after each render call (line 1556). The proposed change must account for the fact that `render()` is called on every interaction and the notice node is moved. The `slot` targeting logic must be re-run on every `render()` call, not just on cascade fire.
**Confidence:** High (this is a correctness fix for a spec violation, not an aesthetic preference)
**Class:** C

### Evolution V4-2 — Pearl appearance threshold: delay until mid-step completion

**Source:** Part 1 F-13 (Low) + Source 5 TurboTax ("Why we ask" placed below complete question) + Mayer coherence principle
**Concrete proposal:** Change the pearl trigger condition in `renderStep1Pearls()` from `state.occlusionType !== null` to `state.lvoLocation !== null || state.mevoLocation !== null`. For `renderStep2Pearls()`: change from `state.time !== null` to `state.time !== null && (state.nihssBucket !== null || state.nihssBasilar !== null || (isMevo() && state.mevoDisabling !== null) || state.time === '>24h')`. This ensures pearls appear after the clinician has made two sequential field selections, not on the first tap. The pearl content (Evidence Landscape, Exclusions, Guideline Update, NIHSS Limitations) is appropriate context once the user has demonstrated they are engaging with the step — not before they have answered the first question.
**Anchoring principle:** Principle 3 (progressive disclosure — make progression mechanics obvious) and Mayer coherence principle (exclude extraneous words and pictures).
**Risk:** Low — purely a timing change. The pearl content itself is unchanged. The only risk is that a user who needs the pearl context on their very first tap (e.g., a first-year resident who doesn't know what LVO means) may not see the "Evidence Landscape" pearl until they have selected an option. Counter-argument: they can always expand the pearl before they proceed; the two-step trigger just prevents layout shift on the first tap.
**Confidence:** Medium (the first-tap layout shift is a real UX problem; the two-step threshold is pragmatic but not research-derived)
**Class:** C (condition change in two render functions — no structural changes, no clinical content changes)

---

### Evolution V4-3 — Conditional field fade-in animation on progressive disclosure

**Source:** Part 2 Source 8 — Laubheimer NN/g 2024 "Branching Questions" study + Principle 7 (explicit feedback)
**Concrete proposal:** When a new category row is added to an active step (e.g., LVO Location appearing after LVO is selected), apply `@keyframes fadeSlideIn { from { opacity: 0; transform: translateY(4px); } to { opacity: 1; transform: translateY(0); } }` with `animation: fadeSlideIn 0.18s ease-out forwards` on the newly-appearing `#field-{fieldKey}` div. Implementation: the render function already replaces inner HTML, so the newly-created DOM elements can receive the animation class via `requestAnimationFrame(() => { newNode.classList.add('field-entered'); })` — but since `innerHTML` replacement re-creates ALL nodes, the animation would trigger on all rows on every render. Better approach: compute a hash of which field IDs are present pre- and post-render; only apply the animation class to IDs that are new. Add the `@media (prefers-reduced-motion: reduce)` guard to skip animation.
**Anchoring principle:** Principle 7 (explicit feedback) — the appearing field is a state change; signaling it prevents user surprise. Laubheimer (2024) specifically found that un-animated conditional field appearance caused errors and anxiety.
**Risk:** Medium — the innerHTML-replacement architecture makes per-node animation non-trivial. Requires either a render-diff algorithm or a refactor to DOM mutation (rather than full innerHTML replacement). This is a Class C effort but the animation quality depends on implementation approach. If done via `innerHTML` naively, ALL rows animate on every re-render (annoying). Recommend deferring this to the JSX implementation where React's reconciliation handles per-node updates natively.
**Confidence:** High (research is directly applicable; implementation risk is architectural, not conceptual)
**Class:** C (in the HTML mockup; inherently resolved in JSX via React reconciliation)

---

### Evolution V4-4 — Inline range validation on ASPECTS / PC-ASPECTS inputs

**Source:** Part 2 Source 7 — GDS inline error messaging + Principle 9 (simple affordances, one-click)
**Concrete proposal:** In `renderNumericInput()` and `setField()`, add a validation check: if the entered value is outside `[min, max]`, render an inline error pill immediately below the input row: `<div class="field-error" role="alert">ASPECTS must be 0–10</div>`. CSS: `.field-error { font-size: 11px; color: #991b1b; background: #fef2f2; border: 1px solid #fecaca; border-radius: 4px; padding: 4px 8px; margin-top: 4px; }`. The interpreter already handles out-of-range values gracefully (returns null or a reasonable result), but the user currently receives no visual feedback that their input is invalid. On mobile, a user who types "12" by accident will see no error until (or unless) the result drawer changes.
**Anchoring principle:** Principle 9 (simple inputs, no extra friction) and Wroblewski (2008) inline validation. GDS confirms: inline validation outperforms post-submit for error recovery.
**Risk:** Low — validation adds a new visual element (error pill) that must fit within the `cat-input-row` layout. The error pill must not shift the next row's position significantly. The `role="alert"` on the error message ensures screen reader announcement.
**Confidence:** High (inline validation is a well-established pattern with strong evidence; the only question is styling, not necessity)
**Class:** C (logic change in `renderNumericInput()` + CSS addition — no clinical content change)

### Evolution V4-5 — Evidence badge links to source on tap

**Source:** Part 2 Source 4 — Glass Health citation pill with external-link icon + Principle 10 (verdict in persistent drawer)
**Concrete proposal:** In the drawer expanded body, the evidence badge (`<div class="evidence-badge ${tbc}">${result.evidenceBadge}</div>`) should become a link to the relevant AHA/ASA 2026 guideline section. The badge renders text like "COR 1 · LOE A" — append a small `→` external-link indicator: `<a class="evidence-badge ${tbc}" href="https://www.ahajournals.org/doi/10.1161/STR.0000000000000375#sec-4-7-2" target="_blank" rel="noopener noreferrer">COR 1 · LOE A →</a>`. The link target should be the exact section cited in `result.evidenceBadge`. For the mockup, one canonical URL per tier can be pre-configured; in the JSX implementation, the `PathwayResult` type can carry a `citationUrl` field.
**Anchoring principle:** Principle 10 (verdict in persistent, unobtrusive drawer) — the drawer is the right place for this citation link. Glass Health's model confirms the badge-as-link pattern. Nielsen H#6 (recognition over recall): clinicians should not have to remember or look up the source separately.
**Risk:** Medium — external links from a clinical tool require `compliance-legal` review per CLAUDE.md §11 agent roster (compliance-legal: activates for auth flows, disclaimers, data disclosure). Flagged as needing compliance review before implementation. The link itself is to a published AHA/ASA guideline, not a commercial page, so risk is low in practice.
**Confidence:** Medium (the pattern is right; compliance gate is the uncertainty)
**Class:** C-clinical (touches clinical content display — the evidence badge is a cited claim surface per `.claude/rules/clinical-surfaces.md`)

### Evolution V4-6 — Pearl-toggle amber icon → slate-500

**Source:** Part 1 F-05 (Medium) + Principle 5 (reserve saturated color for the verdict)
**Concrete proposal:** Change `.pearl-icon { color: #f59e0b; }` to `.pearl-icon { color: #64748b; }`. This is the minimal viable change that eliminates the amber-as-severity collision. **Confidence:** High (the reasoning is grounded in the token system, not subjective preference)
**Class:** B (single CSS property change)
