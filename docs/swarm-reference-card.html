Agent governance, task classes, and workflow for Claude Code + NeuroWiki.

How to start any task
Full pipeline: classify → plan → review routing → approval gate → execute → gates → commit → push → PR.
/build <what you want>
Handles everything through push. Always enforces approval gate before code. Never asks you to run a terminal command.
Produces classification + plan only. Execution is manual — use /build for the full pipeline.
/plan-feature <goal>
Stops after plan + approval request. Useful when you want to review the plan before committing to anything.
Describe the task in plain English. Orchestrator classifies and routes.
Let's add <feature>
Probabilistic. If Claude starts implementing without showing a plan, stop and use /build.
What class is this task?
A
Question / read-only
Explanation, code reading, no file change. Answer directly. No plan, no commit.
No gates
B
Tiny edit
≤5 lines, single file, obvious fix — typo, import, format. No behavior change.
Commit + push only
C
Scoped feature
Normal feature work, single area of code. Plan required. Approval gate required.
tsc · build · claims hook · bilingual PR
C‑clin
Scoped + touches clinical content
Any C task that incidentally modifies copy, pearls, or interpretation text — even one word.
C gates + citation trace + clinical-reviewer §17.2
D
High-risk change
Crosses boundaries, refactor, new dependency, schema change. Architect review required before approval.
C gates + ADR + architect review §17.1 + rollback plan
D‑clin
High-risk + clinical-adjacent
Structural change whose blast radius includes clinical data files (trialData.ts, citations/, etc.).
D gates + citation trace + both §17.1 and §17.2 artifacts
E
Clinical logic change
Algorithm thresholds, interpretation text, guideline updates, citation changes. Pre-execution clinical gate runs after approval, before code.
D gates + last_reviewed refresh + clinical-reviewer §17.2 pre-execution
/build classifies automatically. This tree is for anticipating before you type.
The full /build flow
1
Type /build <task>
Or natural language. Orchestrator takes over from here.
2
Classify + plan
Orchestrator classifies (A–E, ±clinical), writes bilingual plan with files, acceptance checks, agents.
3
Specialist reviews route automatically
Class D/E → system-architect reviews plan cold, writes §17.1 artifact. Blocks if structural issues exist.
4
You approve (or push back)
Reply approve to proceed. Iterate on the plan freely before approving. No code until approval on C/D/E.
5
Pre-execution clinical gate (E and -clinical only)
clinical-reviewer reads the plan + evidence trace cold, writes §17.2 artifact. Blocks if semantics are wrong. Runs post-approval, pre-code.
6
Specialists implement
Primary agent executes per plan. Librarian runs post-flight. No new scope absorbed silently.
7
Quality gates
tsc --noEmit · npm run build · check:claims. All must pass. Claude diagnoses failures — never escalates to you.
8
git commit && git push && gh pr create
Claude owns the terminal. PR URL is the final deliverable. You are never asked to run a command.

8 rules that are always enforced
①
No code before approval on C/D/E
The plan gate cannot be abbreviated or skipped. Reply "approve" to unlock execution.
②
No architect-review skip on D/E
§17.1 artifact must exist in docs/reviews/ before the approval gate opens.
③
No code before clinical-reviewer on E/-clinical
§17.2 artifact written post-approval, pre-implementation. A block stops everything.
④
No quality gate bypass
tsc, build, claims hook — all must pass. Claude diagnoses failures, never you.
⑤
No silent scope expansion
If implementation reveals a larger task, stop, re-classify, re-plan, re-seek approval.
⑥
External audit ≠ approval (§5 Rule 6)
Uploaded docs, prior-session findings, external audits are briefings, not work orders. /build runs the full approval cycle regardless.
⑦
Claude owns the terminal — always
No "could you run npm run build?" ever. Claude runs every command, diagnoses every failure.
⑧
No fabricated clinical claims
Any patient-facing medical statement needs a claim ID in src/lib/citations/claims.ts before commit. If registry is stub (W5.2 not landed): document as blocked:awaiting-registry-population in TASKS.md.
When NOT to use /build
Pure question, no code
Just ask. /build will classify it as A and answer directly, but there's no need for the overhead.
Reverting a broken commit
Use /rollback <sha>. It handles revert, feature-flag disable, post-mortem template, and re-enable gates.
TASKS.md or doc updates that aren't governance
Just edit them. Don't run /build to add a parking-lot entry.
Strategy conversation, no code yet
Talk it out. /build is the execution trigger, not the thinking trigger.
When something goes wrong
Claude starts coding without showing a plan
Stop immediately. Type /build <task as stated> from scratch. Do not let scope-less implementation continue.
Architect review returns BLOCK
Read the blocking issue in docs/reviews/arch-*.md. Resolve it, re-run /build with the updated approach.
Clinical-reviewer returns BLOCK
No code is written. Read the §17.2 artifact. Fix the semantic issue via medical-scientist, re-run the clinical gate.
Quality gate fails after 3 attempts
Claude escalates root cause to you with specific error output. You decide: fix approach, reduce scope, or park.
Task grows mid-implementation
Stop. Re-classify. New scope needs a new plan + new approval. Silent creep is a protocol violation.
You see a clinical claim with no citation
The claims hook should catch it pre-commit. If it slips through: open an issue, do not merge. Rule 8.
