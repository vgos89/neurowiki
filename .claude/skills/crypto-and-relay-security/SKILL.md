# Crypto & Relay Security — NeuroWiki

> On-demand knowledge for `security-engineer`. Also loaded by `system-architect` (when crypto is in scope) and `compliance-legal` (when reviewing disclosure of crypto features).

## Scope

This skill covers the specific cryptographic + transient-data surface NeuroWiki has shipped:

1. Client-side encryption of saved-case bundles before they cross the network
2. Anonymous, time-bounded relay through Supabase with one-time-read semantics
3. Vercel cron with service-role authority and bearer-auth gate
4. Client-side persistence (IndexedDB) of clinically minimal data

It does **not** cover server-side TLS, mTLS, certificate pinning, OAuth/OIDC flows, JWT handling, or hardware-attestation surfaces — none of those are in scope for NeuroWiki today. When they become in scope, extend this skill rather than reasoning from training data.

## Cryptographic parameter floors (as of 2026-05-19)

| Primitive | Current NeuroWiki value | Floor (this skill) | Target | Source |
|---|---|---|---|---|
| PBKDF2 iterations (SHA-256) | 250,000 | 250,000 | 600,000 | OWASP Password Storage Cheat Sheet (2024); NIST SP 800-132 |
| Salt length | 16 bytes | 16 bytes | 32 bytes | NIST SP 800-132 |
| Salt source | `crypto.getRandomValues` | `crypto.getRandomValues` | same | W3C Web Cryptography API |
| AES variant | AES-256-GCM | AES-256-GCM | same | NIST SP 800-38D |
| IV length (GCM) | 12 bytes | 12 bytes | 12 bytes | NIST SP 800-38D §8.2 |
| IV uniqueness | per-record fresh from `crypto.getRandomValues` | per-record fresh | per-record fresh | NIST SP 800-38D §8.3 |
| Authentication tag length | 128 bits (default GCM) | 128 bits | 128 bits | NIST SP 800-38D |

**Floor vs target:** "floor" is the minimum that ships today without a block. "Target" is what the next refresh should adopt. When NeuroWiki's value drops below floor (e.g. iteration counts get raised industry-wide), open a `[P1] Crypto parameter refresh` TASKS.md entry.

**Verifying parameters in the diff:** in `caseCipher.ts`, the iteration count is a numeric literal in the PBKDF2 derivation call. The IV is generated by `crypto.getRandomValues(new Uint8Array(12))` inline. The salt is generated by `crypto.getRandomValues(new Uint8Array(16))` inline. Any deviation from these patterns — especially fixed/constant IV or salt, or `Math.random()` anywhere in the crypto path — is an automatic block.

## Userland vs SubtleCrypto

NeuroWiki uses `window.crypto.subtle` for all cryptographic operations. Any userland implementation (a hand-rolled HMAC, a pure-JS AES, an import of `crypto-js` or `node-forge` in browser code) is an automatic block. SubtleCrypto runs in native code and is constant-time where it matters; userland code in JS is not.

Allowed exceptions: `Math.random()` for non-security UX (animations, jitter). Never for IVs, salts, codes, PINs, or anything authentication-adjacent.

## Row Level Security — patterns and red flags

NeuroWiki's `transfers` table holds encrypted case bundles. RLS policies must enforce:

1. **Anonymous insert allowed, but only with the required fields populated.** `WITH CHECK` clause on insert ensures the payload, code, and TTL are all set.
2. **Anonymous select allowed, but only by the row's `code` (which the requester must know to even pose the query).** This is the "knowledge-of-code is the auth" pattern; the encryption-on-top-of-code is what makes this safe.
3. **Anonymous delete allowed, but only on the row the client just selected.** This is the one-time-read semantics — the receiver fetches and deletes in a single transaction.
4. **Service-role can read/write/delete anything.** Used only by the cron and only from server-side code.

**Red-flag policies:**
- `USING (true)` on `SELECT` for anonymous role — opens enumeration attacks unless paired with a non-guessable lookup key.
- No `WITH CHECK` on `INSERT` — allows malformed payloads to land and break the cleanup cron.
- Service role exposed to the browser bundle — automatic block; the service role belongs only in Vercel functions.
- Anon role with `UPDATE` permission on transient table — there is no legitimate update path; anonymous updates are a foothold.

**Reviewing a new RLS migration:** check that every policy named in the migration (a) has a `USING` clause OR `WITH CHECK` clause OR both, (b) names the correct role (`anon` vs `authenticated` vs `service_role`), (c) does not contradict an existing policy on the same table for the same role.

## Transient-relay design — NeuroWiki data-flow

```
Sender                 Supabase transfers           Receiver
------                 -------------------          --------
1. Pick 4-digit PIN
2. Encrypt cases with PIN (PBKDF2 → AES-256-GCM)
3. Generate 6-digit code
4. POST insert with code + ciphertext + iv + salt + expires_at (now + 15min)
                       INSERT (anon, WITH CHECK)
                       row {code, ciphertext, iv, salt, expires_at}
5. Show code + PIN to sender via QR/text/voice
                                                    6. Receiver enters code
                                                    7. GET select by code (anon)
                       SELECT (anon, USING code match + not expired)
                                                    8. Receiver enters PIN
                                                    9. Decrypt (PBKDF2 → AES-256-GCM)
                                                    10. Persist locally in IndexedDB
                                                    11. DELETE by code (anon, USING code match)
                       DELETE (anon, USING code match)
                       row gone — one-time-read confirmed
```

**Cleanup cron (independent of the read path):**

```
Vercel cron @ 0 4 * * * UTC
  → /api/transfer-cleanup
  → Authorization: Bearer ${CRON_SECRET} (constant-time compare server-side)
  → service-role client: DELETE FROM transfers WHERE expires_at < now()
```

This is the belt-and-suspenders TTL: even if a receiver never reads, rows are reaped within ~24h.

**Reviewing changes to this flow:** any change that (a) widens TTL beyond 15 minutes, (b) removes the delete-on-read step, (c) makes the cron's auth check non-constant-time, (d) exposes the service-role key to the client, or (e) decouples encryption parameters from the receiver's PIN — automatic block.

## Code / PIN entropy

- **6-digit code:** 10^6 = ~20 bits. Adequate for a single-use, 15-minute, rate-limited lookup. **If no rate-limit exists at the Supabase layer, surface as a finding** — without rate-limiting, 20 bits is brute-forceable in under 30 seconds.
- **4-digit PIN:** 10^4 = ~13 bits. Adequate only when paired with the code (the attacker would need to brute-force both, and the row vanishes after one successful read). Standalone, 13 bits is trivially brute-forceable.
- **Combined surface:** ~33 bits with TTL of 15 minutes. Reasonable, but the system depends on the one-time-read enforcement to make brute-force pointless. If one-time-read is ever broken (e.g. delete happens after instead of inside the read transaction), the security model collapses.

**Reviewing changes to code/PIN sizing:** shrinking either dimension below current = block. Removing the one-time-read coupling = block. Replacing the PIN-derived key with a static or wire-derived key = block.

## Secret handling

NeuroWiki secrets:

| Secret | Storage | Classification | Rotation cadence (proposed) | Exposure surface |
|---|---|---|---|---|
| `SUPABASE_SERVICE_ROLE_KEY` | Vercel env (Sensitive) | Highest — full DB access | 90 days | Server only (`api/transfer-cleanup.ts`, never client) |
| `CRON_SECRET` | Vercel env (Sensitive) | High — cron-trigger auth | 180 days | Server only |
| `VITE_SUPABASE_ANON_KEY` | Vercel env (public) | Public by design | n/a (anon key is intended for client) | Client bundle (intentional) |
| `VITE_SUPABASE_URL` | Vercel env (public) | Public | n/a | Client bundle |

**Reviewing changes to env config:**

- Any new secret should be classified explicitly (sensitive vs public).
- `VITE_*` prefix means the variable lands in the client bundle — anything sensitive must NOT carry the `VITE_*` prefix.
- A diff that moves a secret from server-only to `VITE_*` is an automatic block.
- A diff that adds a new server-side secret should also add a rotation cadence note (to ADR or runbook).

**Auth check pattern:**

```typescript
// api/transfer-cleanup.ts
const auth = req.headers.authorization;
const expected = `Bearer ${process.env.CRON_SECRET}`;
// constant-time compare — important
if (!auth || !timingSafeEqual(Buffer.from(auth), Buffer.from(expected))) {
  return res.status(401).json({ error: 'unauthorized' });
}
```

Naive `auth === expected` comparison is a timing-attack surface; flag if seen.

## Client-side persistence (IndexedDB)

**Schema versioning:** `SAVED_CASE_SCHEMA_VERSION` in `src/lib/cases/types.ts`. Bumps must be additive when possible; structural migrations require:
- A versioned upgrade path in `openDB()` (the `onupgradeneeded` handler must handle each version delta)
- Forward-compatibility test (v(N) reader on v(N+1) data → safe degrade)
- Backward-compatibility test (v(N+1) reader on v(N) data → reads correctly)

**Plaintext exposure:** IndexedDB content is plaintext on disk. NeuroWiki's clinical-minimal rule (initials-only, no PHI fields) is the mitigation. Any new field added to `SavedCaseData` that could carry PHI is reviewed — initials are deliberately the only identifier.

**Reviewing a new SavedCaseData field:**
- Could this field carry PHI? (full name, MRN, DOB, exact LKW timestamp paired with initials in a way that re-identifies)
- If yes — block or require explicit redaction strategy.
- If no — confirm in the security review notes and approve.

## Service worker (when shipped)

NeuroWiki ships a service worker via `vite-plugin-pwa`. Review on every change to:
- The PWA plugin config in `vite.config.ts`
- The `runtimeCaching` rules
- The registration code

**Hard rules for service-worker security:**
- `/api/*` must be `NetworkOnly` — never cache an authenticated or stateful endpoint
- No caching of responses that include `Authorization` or `Cookie` headers
- Update strategy is `autoUpdate` with user notification — never silent swap during a clinical workflow
- Cache scope is the origin only — never proxy third-party origins

## Common findings library

| Finding | Severity | Recommended action |
|---|---|---|
| PBKDF2 below 250k iterations | Block | Raise to 600k target; coordinate decryption-compatibility window |
| IV reused across encryptions | Block | Replace with per-record `crypto.getRandomValues` IV |
| `Math.random()` in crypto path | Block | Replace with `crypto.getRandomValues` |
| Service role key in `VITE_*` env | Block | Remove from client; rotate immediately |
| RLS `USING (true)` on transient SELECT without code-match | Block | Add `code = current_setting('request.code', true)` or equivalent |
| No `WITH CHECK` on transient INSERT | Advisory | Add `WITH CHECK (length(code)=6 AND expires_at > now())` |
| Cron auth using `===` instead of constant-time | Advisory | Use `timingSafeEqual` |
| Naive password storage (without PBKDF2/Argon2/bcrypt) | Block | n/a today — flag if introduced |
| New `SavedCaseData` field that could carry PHI | Block | Reject field; surface to clinical-reviewer |
| Schema bump without migration story | Advisory | Document additive vs structural; add upgrade-path test |
| Service-worker caches `/api/*` | Block | Move endpoint to `NetworkOnly` |
